### Overview

The AI Semantic Caching policy reduces redundant backend calls to LLM APIs by recognizing semantically similar requests using vector embeddings. When a request matches a cached vector within a configured similarity threshold, the cached response is returned without invoking the backend. This reduces token consumption, improves response latency, and lowers operational costs for LLM-based APIs.

### Key concepts

#### Semantic similarity matching

Traditional caching uses exact string matching. Semantic caching uses vector embeddings to identify requests with similar meaning. For example, "What is the capital of France?" and "Capital of France?" are treated as equivalent. The policy generates an embedding for each request prompt, queries the vector store for similar cached embeddings, and returns the cached response if the similarity score exceeds the configured threshold.

#### Vector embeddings

A vector embedding is a numerical representation of text content generated by a machine learning model. The policy supports three embedding providers: ONNX (local inference), OpenAI (cloud API), and custom HTTP endpoints. Embeddings are stored in a vector store alongside cached response metadata (headers, status, body, token count).

#### Cache metadata and filtering

Each cached vector includes metadata extracted from the request context using EL expressions. Metadata keys can be encoded using MurmurHash3 for privacy. This enables context-aware caching (e.g., per-API, per-tenant) by filtering vector store queries based on metadata values.

### Prerequisites

- Gravitee API Management 4.x or later with Enterprise Edition license
- One of the following vector store backends configured:
  - Redis with vector search support
  - AWS S3 with vector indexing enabled
- Access to an embedding model provider:
  - ONNX runtime for local inference (Xenova models)
  - OpenAI API credentials
  - Custom HTTP endpoint returning embeddings
- API plan configured with the AI Semantic Caching policy

### Creating an AI Semantic Caching configuration

Configure the policy on an API plan by referencing the text embedding model and vector store resources by name. Define a `promptExpression` to extract the content to embed (e.g., the last user message in a chat completion request). Set a `cacheCondition` to control which responses are cached (e.g., only successful responses with status 200-299). Optionally add metadata parameters to enable context-aware caching. For example, to cache responses per API, add a parameter with `key: "api"`, `value: "{#context.attributes['api']}"`, and `encode: true` to hash the API identifier.

### Creating a subscription with metadata filtering

When multiple APIs share the same vector store, use metadata parameters to isolate cached responses. Configure the policy with a parameter that extracts a unique identifier from the request context (e.g., API ID, tenant ID, or user ID). Set `encode: true` to hash the value using MurmurHash3 with Base64 URL-safe encoding. The vector store query will filter results to only those matching the encoded metadata value, preventing cache collisions across contexts.

### Client configuration

No client-side configuration is required. The policy operates transparently at the gateway level. Clients send requests as usual, and the gateway returns cached responses when semantic similarity is detected. Response headers and body are identical to the original backend response.

### Architecture notes

#### Policy execution flow

The policy executes during the `onRequest` phase. It extracts the prompt content using the configured EL expression, generates an embedding via the text embedding model resource, builds metadata from configured parameters (with optional encoding), and queries the vector store for similar cached vectors. If a cache hit is detected (similarity score exceeds threshold), the policy replaces the default HTTP invoker with a custom invoker that returns the cached response. If a cache miss occurs, the custom invoker calls the backend and caches the response.

### Configure AI Semantic Caching

Configure the AI Semantic Caching policy on an API plan by referencing the text embedding model and vector store resources by name.

#### Configuration steps

1. Add the AI Semantic Caching policy to an API plan.
2. Set the `modelName` property to the name of your configured text embedding model resource (e.g., `ai-model-text-embedding-resource`).
3. Set the `vectorStoreName` property to the name of your configured vector store resource (e.g., `vector-store-redis-resource`).
4. Define the `promptExpression` using EL to extract the content to embed. Example:
   ```
   {#jsonPath(#request.content, '$.messages[-1:].content')}
   ```
   This expression extracts the last user message from a chat completion request.
5. Set the `cacheCondition` using EL to control which responses are cacheable. Example:
   ```
   {#response.status >= 200 && #response.status < 300}
   ```
   This condition caches only successful responses with status codes 200-299.
6. (Optional) Add metadata parameters for context-aware caching. Example:
   ```json
   [{"key": "api", "value": "{#context.attributes['api']}", "encode": true}]
   ```
   This parameter isolates cached responses when multiple APIs share the same vector store.

### Metadata encoding

When a parameter's `encode` property is set to `true`, the value is hashed using MurmurHash3 (128-bit) and encoded as Base64 URL-safe without padding. This ensures metadata values are stored in a fixed-length, privacy-preserving format suitable for vector store filtering.

### Cache eviction

If `allowEviction` is enabled in the vector store configuration, each cached vector includes an `expireAt` timestamp calculated as `Instant.now().plus(evictTime, evictTimeUnit.toChronoUnit())`. The vector store backend is responsible for removing expired entries based on this metadata field.

### Restrictions

- Requires Gravitee API Management Enterprise Edition license
- Policy is published to `graviteeio-ee/apim/plugins` path
- Embedding model resource must be configured before policy can be applied
- Vector store resource must be configured before policy can be applied
- OpenAI embedding model requires valid API key and organization/project IDs (if applicable)
- AWS S3 vector store requires valid AWS credentials or IAM role with S3 access
- Redis vector store requires Redis instance with vector search support enabled
- Embedding dimensions must be non-negative if specified for OpenAI provider
- Vector entity ID must not be null or blank
