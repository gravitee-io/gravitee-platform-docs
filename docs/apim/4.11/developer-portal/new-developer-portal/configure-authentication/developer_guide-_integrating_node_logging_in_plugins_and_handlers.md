# Developer Guide: Integrating Node Logging in Plugins and Handlers

## Developer Integration

This section explains how plugin and handler developers integrate node logging infrastructure into custom code to enable MDC enrichment and context-aware logging in reactive Gateway handlers.

### Prerequisites

* `gravitee-node` 8.0.0-alpha.2 or later
* `gravitee-gateway-api` 5.0.0 or later
* Write access to project source code and `pom.xml`
* Understanding of reactive programming patterns in Gravitee Gateway

### Replacing LoggerFactory with NodeLoggerFactory

Developers must replace `org.slf4j.LoggerFactory` with `io.gravitee.node.logging.NodeLoggerFactory` to enable MDC enrichment. Direct use of `LoggerFactory` prevents the logging infrastructure from populating contextual keys such as `nodeId`, `apiId`, and `application`.

1. Update imports in all classes that declare loggers:
   * Replace `import org.slf4j.LoggerFactory;` with `import io.gravitee.node.logging.NodeLoggerFactory;`
2. Update logger declarations:
   * Replace `LoggerFactory.getLogger(MyClass.class)` with `NodeLoggerFactory.getLogger(MyClass.class)`
3. For projects using Lombok, configure the `@CustomLog` annotation:
   * Add the following line to `lombok.config` in the project root:
     ```
     lombok.log.custom.declaration = org.slf4j.Logger io.gravitee.node.logging.NodeLoggerFactory.getLogger(TYPE)
     ```
   * Replace `@Slf4j` annotations with `@CustomLog` in class declarations

ArchUnit rules enforce this pattern. Builds fail if direct `LoggerFactory` usage is detected in non-exempt classes.

### Using Context-Aware Logging in Reactive Handlers

In reactive Gateway handlers, methods that accept an `ExecutionContext` parameter must use `ctx.withLogger(log)` instead of direct logger calls. This pattern ensures that MDC keys are populated from the execution context before each log statement.

1. Identify methods that accept an `ExecutionContext` parameter and call logger methods such as `log.info()` or `log.debug()`
2. Replace direct logger calls:
   * Change `log.info("message")` to `ctx.withLogger(log).info("message")`
   * Change `log.debug("message", arg)` to `ctx.withLogger(log).debug("message", arg)`
3. The `withLogger()` method returns a lazy-initialized context-aware logger that populates MDC from the execution context before each log statement

ArchUnit rules scan packages including `io.gravitee.gateway.reactive.handlers`, `io.gravitee.gateway.reactive.core`, and `io.gravitee.apim.plugin`. Builds fail if direct logger calls are detected in methods with `ExecutionContext` parameters.

### ArchUnit Enforcement Rules

The `gravitee-archrules-maven-plugin` enforces logging architecture rules via two Maven goals:

* **`global-logging-check`**: Validates that classes use `NodeLoggerFactory` instead of `LoggerFactory`
* **`execution-context-logging-check`**: Validates that reactive handler methods use `ctx.withLogger()` instead of direct logger calls

Both goals run during the `verify` phase. Classes with the suffix `ConfigurationEvaluator` are exempt from enforcement because they are generated by `gravitee-plugin-annotation-processors`.

ArchUnit checks are disabled in CI builds (`-Dgravitee.archrules.skip=true`) to reduce build time. Developers must run `mvn verify` locally to validate logging rules before committing code.

### Restrictions

* **Plugin migration**: Not all plugins have been migrated to use `NodeLoggerFactory` as of APIM 4.11. Some logs may lack contextual information until the migration is complete in APIM 4.12.
* **Hierarchy cache**: The execution context hierarchy cache is never cleared. In hot-reload scenarios, this may cause stale class references.
